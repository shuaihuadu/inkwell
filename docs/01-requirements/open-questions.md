## OQ-001 数据出境与合规边界

- **问题**：v1 是否允许把用户对话内容、上传的图片 / 语音 / 文档原文，发送给云端模型厂商（Azure OpenAI / OpenAI / Claude / Qwen / 智谱 …）？是否需要做敏感字段脱敏？是否禁止上传特定类型的内容（如 PII、商业机密、源代码）？
- **为什么需要答**：本项目用户原话是"对话历史全存到后端，满足合规要求"，但具体合规要求是哪一类（GDPR / 等保 / 金融 / 医疗 / 公司内部红线？）未给出。这条决策直接决定后端是否要做内容分类、脱敏管线、是否禁用某些模型厂商。不答会让 H2 / H3 的安全设计悬空。
- **影响范围**：REQ-005 / REQ-009 / REQ-013 / REQ-016 / EX-002 / 数据边界 §8。
- **候选答**：
  - **A**. 公司已与所有候选模型厂商签 DPA，对话与文件可原文发送，**不做脱敏**。后果：实现成本最低；合规责任全部前移到厂商合同与人工审核。
  - **B**. 可发送，但需对**人名 / 电话 / 邮箱 / 身份证 / 银行卡**等 PII 字段做客户端预脱敏。后果：需引入脱敏管线；脱敏会影响 Agent 回答准确性，需在 trace 中保留原文 vs 脱敏后双轨。
  - **C**. 只能向**特定白名单**模型厂商发送（如仅 Azure OpenAI），其他厂商默认禁用，由管理员逐项放开。后果：UI 需做"模型可用性"展示；多家云服务的接入位仍要做（REQ-005），但默认禁用。
  - **D**. 不确定 / 这是公司政策，先记 OQ 让法务回答。
- **回答**：

  > 公司已与所有候选模型厂商签 DPA，对话与文件可原文发送，不做脱敏。后果：实现成本最低；合规责任全部前移到厂商合同与人工审核。

## OQ-002 v1 是否给出最低性能 / 可用性底线

## OQ-003 多模态语音输入的 ASR 路由策略

- **问题**：REQ-016 要求 v1 支持语音输入，但 ASR（自动语音识别）走哪条路径未定：（a）作为后端独立服务（如 Azure Speech / 自研 Whisper），（b）作为模型本身的多模态能力（如 GPT-4o audio）。两条路径对客户端协议（流式 vs 一次性）、计费、合规都有不同影响。
- **为什么需要答**：H2 选型 / H3 设计 API 协议时无法二选一。
- **影响范围**：REQ-016、§5.2 后端配置项、EX-004 多模态兼容、OQ-001。
- **候选答**：
  - **A**. 后端独立 ASR（v1 优先 Azure Speech，与 Azure OpenAI 同生态）；语音先转文本再注入对话，所有模型都能用。后果：开发简单；丢失语音原始信息（情绪 / 重音）。
  - **B**. 直接用支持音频的多模态模型（如 GPT-4o audio）；不接 ASR。后果：仅在选支持音频的模型时可用；其他模型语音输入被拒。
  - **C**. 两路并存，按所选模型自动路由。后果：实现复杂度最高；UI 需展示能力差异。
  - **D**. 不确定 / 让 H2 决定。
- **回答**：

  > 选 A。后端独立 ASR（v1 优先 Azure Speech，与 Azure OpenAI 同生态）；语音先转文本再注入对话，所有模型都能用。接受后果：开发简单；丢失语音原始信息（情绪 / 重音）。

- **决策日期**：

  > 2026-05-08

- **决策人**：

  > Inkwell / Owner

- **卡点等级**：closed 2026-05-08
- **回写**：requirements.md §5.1 REQ-016 实现说明·语音、§5.2 后端配置项（新增 Azure Speech）、§9 上游决策与假设第 19 条。

## OQ-005 v1 账号开通方式：管理员后台创建的具体形态

- **问题**：用户原话"账号体系先使用最简单的，但是未来需要开放注册、接入其他SSO和互联网公开的OAuth"。"最简单"具体指什么形态？影响 H2 是否引入 IAM 抽象层（决定后续接入 SSO / OAuth 的难度）。
- **为什么需要答**：H2 / H3 是按"明文用户名密码 + 一张用户表"做，还是一开始就引入 OAuth2 / OIDC 抽象层。
- **影响范围**：REQ-001、§3 用户角色、§7 权限边界、§10 不做范围（开放注册 / SSO 推迟）。
- **候选答**：
  - **A**. v1 用户名 + 密码；账号由后端管理员通过 SQL / 管理脚本创建；不引入抽象层（v2 再重构）。后果：v1 最快；v2 引入 SSO 时要重做用户表与登录链路。
  - **B**. v1 用户名 + 密码，但**走 OAuth2 / OIDC 内嵌 IdP**（如内置一个最小 IdP 服务），让客户端始终走 OAuth；v2 接外部 IdP 时只换 IdP 不换链路。后果：v1 复杂；演进平滑。
  - **C**. v1 用户名 + 密码 + 引入用户提供商抽象（不真上 OIDC，但代码层留下接口）。后果：折中。
  - **D**. 不确定 / 让 H2 决定。
- **回答**：

  > 选 A。v1 用户名 + 密码；账号由后端管理员通过 SQL / 管理脚本创建；不引入抽象层（v2 再重构）。接受后果：v1 最快；v2 引入 SSO 时要重做用户表与登录链路。

- **决策日期**：

  > 2026-05-08

- **决策人**：

  > Inkwell / Owner

- **卡点等级**：closed 2026-05-08

## OQ-007 团队角色"暂不区分"的下游含义

- **问题**：v1 仅一类 Member 角色，"系统管理员"由后端运维直接操作（不在产品内体现）。若团队规模到 100 人，没有产品内的"团队所有者"角色，谁来仲裁"撤销共享 / 删除他人共享 Agent / 关闭异常账号"等运营事务？是否接受由后端运维（运行 SQL）兜底？

## OQ-008 长期记忆策略与对话历史持久化的协同

- **问题**：REQ-010 要求长期记忆（开 / 关 / 摘要式），NFR-005 要求对话历史全量存后端。两者关系是：长期记忆是从历史中**摘要 / 抽取的衍生数据**，还是独立的另一份数据？记忆能否跨 Agent 共享？关闭长期记忆是否同时**删除已摘要内容**？

  - **C**. 不确定 / 让 H3 决定。
- **回答**：

  > 选 C。该设计点推迟到 H3，由详细设计中的数据表设计决定。

  > 2026-05-08

## OQ-010 v1 是否需要"团队共享 Agent 库"独立于"我的 Agent"

- **影响范围**：REQ-002、ui-spec 主界面布局。
- **候选答**：
  - **A**. 一个混合列表，按 Owner 列与"是否团队共享"标签区分。后果：UI 简单；列表可能拥挤。
  - **B**. 两个 tab："我的 Agent" / "团队共享"。后果：UI 清楚；切换成本。
  - **C**. 三档 tab："我的" / "团队共享" / "我使用过"。后果：发现性最好；信息架构最复杂。

- **问题**：NFR-002 要求 Win11 + macOS 12+ Apple Silicon 功能等价。**键盘快捷键**在两端是否要保持一一对应（macOS `Cmd` ↔ Windows `Ctrl`）？是否在 v1 提供"快捷键"设置面板让用户自定义？
