# 自定义 Agent 功能 · H1 需求草稿

> 本草稿基于三轮结构化问询产出。第二轮访谈（2026-05-06）关闭了 [open-questions.md](./open-questions.md) 中的初版 OQ-001~015；第三轮访谈（2026-05-06）应客户硬约束**重启** OQ-001 与 REQ-006 的 vNext 决策，将 Skill / Tool / MCP 占位提升到 MVP（详见第 13.3 节与 OQ-022~029）。本草稿仍处 `draft` 状态——`status` 字段需在 UI 说明 / 原型 / 评审记录补齐后才转 `reviewed`。

## 1. 项目背景

Inkwell 当前仓库内除工程骨架（`.harness-engineering/` 与 `.github/`）外尚无产品代码或既有需求文档；本草稿是仓库内**首条** H1 需求凭证（REQ-001 起步）。

本特性的目标：让无技术背景的业务用户能够在不依赖工程介入的前提下，自助定义、试运行、管理属于自己的 Agent，并按需要把它分享给他人使用。

> 本草稿不替用户选择前端框架、后端语言、数据库、向量库、模型网关或部署形态——这些属于 H2 的工作。

## 2. 目标用户

- **核心目标用户**：Inkwell 产品的终端业务用户。
  - 无技术背景，不熟悉 prompt 工程、模型参数语义、工具 schema、向量库 / MCP / 知识库索引等开发者概念。
  - 期望"像设置一个个人助手"那样定义 Agent。
- **非目标用户**（不在本特性服务范围）：
  - 平台运维与基础设施管理员
  - Inkwell 自身代码层 Agent 框架的开发者
  - 第三方插件 / Agent 提供商

## 3. 用户角色

第二轮访谈关闭 OQ-003 = A、OQ-004 = A 后，MVP 锁定**两类**角色，**不**引入团队角色，**不**引入匿名访客。

| 角色         | 说明                                                                                       |
| ------------ | ------------------------------------------------------------------------------------------ |
| Agent 作者   | 已登录用户。可创建、编辑、试运行、复制、删除属于自己的 Agent。                             |
| Agent 使用者 | 已登录用户。可浏览自己有权访问的 Agent 列表，并对其发起对话。同一人可同时是作者与使用者。 |

> "已登录用户"是平台层依赖（详见 12.2 节 D2 与 12.1 节 R7）。本特性不内置登录子系统。

## 4. 核心场景

| 编号 | 场景                            | 描述                                                                                                                       |
| ---- | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| S1   | 从零创建一个新 Agent            | 用户登录后，新建一个 Agent，给它取名、写一段角色设定，立刻在试运行界面跟它对话验证效果，满意后保存。                       |
| S2   | 在已有 Agent 上微调             | 用户从自己的 Agent 列表打开一个已有 Agent，调整名称 / 头像 / Instructions，再次试运行验证后保存。                           |
| S3   | 复制一个 Agent 作为起点         | 用户复制自己已有的某个 Agent，得到一个新的可独立编辑的副本，用于派生新的业务变体。                                         |
| S4   | 删除不再需要的 Agent            | 用户删除自己不再需要的 Agent；删除后该 Agent 在列表与所有对话入口立即不可达；7 天内可恢复（详见 DB-004），过期硬删除。     |
| S5   | 边改边试                        | 用户在编辑过程中能随时切到试运行视图发起对话，验证当前修改的 Prompt / 元数据是否符合预期，再决定是否保存。                 |
| S6   | （vNext）把 Agent 给别人用      | 用户把 Agent 共享给其他登录用户使用——分享按固定快照锚定，作者后续修改不影响已分享版本。本期不上线。                       |
| S7   | （已移除）看 Agent 跑得怎么样   | 已移除：运行时观测由平台运维模块承担，不在本特性内（详见 ND-007）。                                                        |
| S8   | （vNext）把 Agent 带走 / 带进来 | 用户能把 Agent 配置导入 / 导出，仅支持本平台 JSON 格式，不兼容外部 schema。本期不上线。                                    |
| S9   | 给 Agent 装一项 Skill           | 用户从平台 Skill 库勾选一个 Skill（如客服话术 / 翻译 / 邮件起草），或在线编辑 / 导入符合 [agentskills.io](https://agentskills.io/specification) 标准的 SKILL.md，使 Agent 在对话中具备该专长。 |
| S10  | 给 Agent 启用内置工具           | 用户在工具列表里勾选"联网搜索""当前日期"等平台内置工具，Agent 在试运行时按需调用；首次启用每个工具时弹"用户须知"明示数据外发。 |

## 5. 功能范围

每条需求项标注**分层标签**：

- **MVP**：本期必须落地；验收标准必须为黑盒可机械验证。
- **vNext**：能力完整闭环所需，但本期不上线；本节仍记录其"vNext 阶段须遵循的关键决策"，避免下一轮重启同样的讨论。

> 本节**不**约定任何 UI 形态、字段拆分、数据结构、API 形状——这些归 H1 的 UI 说明、H2 与 H3。本节只回答"做什么 / 为谁做 / 怎么验"。

### REQ-001 Agent 生命周期管理（创建 / 列表 / 复制 / 重命名 / 删除）—— MVP

**说明**：用户能在自己的视图下完整管理属于自己的 Agent。

**验收标准**：

- AC-001-1 已登录用户进入"我的 Agent"视图能看到一份列表，且默认按"最近更新时间"倒序排列。
- AC-001-2 用户可以执行"新建 Agent"操作，操作成功后该 Agent 立即出现在列表顶部。
- AC-001-3 用户可以对自己创建的某个 Agent 执行"复制"，操作成功后列表中出现一份新副本，副本归当前用户所有，名称在原名基础上自动加可识别后缀。
- AC-001-4 用户可以重命名自己的 Agent；重命名后列表中显示的名称在保存动作完成后立即与新名称一致。
- AC-001-5 用户可以删除自己创建的 Agent；删除后该 Agent 在该用户的列表中不再出现，对所有原本可达入口（直链 / 对话入口）立即不可达；删除采用软删除，保留 7 天供恢复（详见 DB-004），过期硬删除。
- AC-001-6 列表只展示"当前用户拥有"的 Agent；MVP 不展示"被分享给当前用户"的 Agent（共享能力 REQ-008 = vNext）。
- AC-001-7 MVP 不实现"启用 / 停用"开关。

### REQ-002 Agent 基础元数据（名称 / 头像 / 描述）—— MVP

**说明**：用户能为 Agent 设置可被使用者识别的最小身份信息。

**验收标准**：

- AC-002-1 名称为必填字段；为空时保存动作必须被拒绝并给出提示。
- AC-002-2 描述为选填字段；允许为空。
- AC-002-3 头像为选填；未上传时 Agent 在所有展示位均使用平台默认占位头像。
- AC-002-4 用户保存名称 / 头像 / 描述任一字段后，列表（REQ-001）与试运行（REQ-004）中展示的对应内容必须与新值一致。
- AC-002-5 名称长度上限 ≤ 50 字符（含空格），允许 emoji，允许中英文及其他多语言混排（OQ-014 = A）。
- AC-002-6 头像格式 / 大小上限由 H2 / H3 设计阶段在平台公共上传规范内细化；本 REQ 不另立约束。
- AC-002-7 头像 / 描述不做事前 / 事后内容审核（OQ-012 = A，相关合规风险见 12.1 节 R3）。

### REQ-003 Instructions（系统指令）—— MVP

**说明**：用户能为 Agent 写一段决定其“性格 / 任务 / 风格 / 边界”的指令文本（UI 标签为 `Instructions`，2026-05-06 评审会决议），并在试运行时被使用。本 REQ 不约束指令是单字段还是多字段；具体字段拆分（如“人设 / 任务 / 风格 / 红线”）属于 H1 的 UI 说明范围。

**验收标准**：

- AC-003-1 用户能在 Agent 详情页保存一段 Instructions；保存动作必须给用户明确成功 / 失败反馈。
- AC-003-2 用户保存后下一次试运行（REQ-004）必须使用该 Instructions 作为 Agent 的系统指令。
- AC-003-3 用户在编辑过程中可放弃当前修改回到最近一次已保存版本。
- AC-003-4 Instructions 允许中英文及多语言；不做关键词审核（OQ-012 = A）；具体字符长度上限由 H2 / H3 在网关 / 模型 token 预算约束下细化，不在本 REQ 写死。

### REQ-004 在线试运行 / 对话调试 —— MVP

**说明**：作者在编辑过程中可直接发起对话验证 Agent 行为。这是非技术用户判断 Agent 是否可用的**唯一可达路径**——没有它就无法闭环。

**验收标准**：

- AC-004-1 用户在 Agent 详情 / 编辑页能进入"试运行"对话界面，无需独立部署或额外发布步骤。
- AC-004-2 试运行使用当前最近一次已保存的元数据（REQ-002）和 Instructions（REQ-003）；编辑后未保存的修改不应进入试运行（除非 H1 UI 说明明确支持“用未保存的临时副本试运行”）。
- AC-004-3 用户能中止正在进行的对话、清空当前对话、重新开始一段新对话。
- AC-004-4 模型调用失败 / 超时时，平台自动重试最多 3 次（每次间隔由 H2 / H3 在网关层细化，建议 ≤ 2 秒），3 次仍失败时向用户展示明确的"出错了，请稍后重试"提示，本次失败调用不计费（OQ-010 = B）。
- AC-004-5 试运行所产生的对话记录留存策略：默认仅作者本人可回看；不进入运行时观测样本；不被平台用于训练（OQ-008 = A）；保留时长由 H2 / H3 在 7 天软删除窗口（DB-004）下细化。

### REQ-005 模型选择与运行参数 —— vNext

**vNext 阶段必须遵循的关键决策（已锚定，禁止重启讨论）**：

- 模型清单首批 = `gpt-4.1`（OpenAI 单一供应商；本期网关只需适配 OpenAI SDK；OQ-007）。
- Key 来源 = 平台托管，配置写入系统配置文件，不向终端用户暴露（OQ-007）。
- 配额单位 = 每用户每天 N 次调用（具体 N 值由 vNext 阶段产品 / 运营定）。
- 计费形态 = 按 token 计费（OQ-007）。
- UX 形态 = 模板 / 向导（"更稳 / 更准 / 更有创意"等业务化预设），**不**向终端用户暴露 temperature / top_p / max_tokens 等开发者参数（OQ-002 = A，NFR-001）。

**MVP 影响**：MVP 阶段所有 Agent 一律走平台预设模型（即 `gpt-4.1` + 默认参数），用户不可调；本 REQ 不阻塞 MVP 上线。

### REQ-006 知识库挂载 —— vNext / pending

**状态**：本期不上线；后续是否换入本特性待 [OQ-019](./open-questions.md) 关闭后决定（第三轮访谈中明确 "pending"）。

**vNext 阶段必须遵循的关键决策**：

- UX 形态 = 模板 / 向导型；如挂载点必须暴露给非技术用户，必须包装成业务化预设（OQ-002 = A）。
- 知识库内容合规 = 不被用于训练，仅在用户所选地域驻留（OQ-008 = A）。
- 文件格式 / 单文件大小 / 累计上限待 OQ-019 关闭后细化。

> 说明：原 REQ-006 拆为三个独立项。Skill 拆出为 [REQ-010](#req-010-skill-管理agentskillsio-标准) MVP；Tool 拆出为 [REQ-011](#req-011-内置-tool-启用) MVP；MCP 拆出为 [REQ-012](#req-012-mcp-server-集成-mvp-ui-占位--vnext-后端) MVP UI 占位 + vNext 后端。本 REQ-006 仅保留为知识库专项。

**MVP 影响**：MVP 不提供知识库挂载入口；不阻塞 MVP 上线。

### REQ-007 版本管理与导入 / 导出 —— vNext

**vNext 阶段必须遵循的关键决策**：

- 导入 / 导出格式 = 仅本平台 JSON，**不**兼容外部 schema（OQ-006 = A）。
- 版本字段 = 必填——分享语义按固定快照锚定（OQ-009 = B），无版本号即无法实现"快照不变"。

**MVP 影响**：MVP 不提供导入 / 导出能力；不阻塞 MVP 上线。

### REQ-008 共享与权限（私有 / 公开）—— vNext

**vNext 阶段必须遵循的关键决策**：

- 可见域 = 仅"私有 / 公开"两档，**不**做团队（OQ-003 = A）。
- "公开" = 仅平台已登录用户可见；**不**开放互联网匿名访问（OQ-004 = A）。
- 分享语义 = 固定快照——作者改 / 删 Agent 不影响已分享版本，使用者侧版本号必须在 UI 上明示（OQ-009 = B）。
- 内容审核 = 不做事前 / 事后审核；相关合规风险作为已知接受项登记到 12.1 节 R3（OQ-012 = A）。

**MVP 影响**：MVP 阶段所有 Agent 一律私有（PB-001、AC-001-6）；不阻塞 MVP 上线。

### REQ-010 Skill 管理（agentskills.io 标准）—— MVP

**说明**：用户能为自己的 Agent 选用 / 创建 Skill——采用开放 [agentskills.io 标准](https://agentskills.io/specification) 的 SKILL.md 格式。本 REQ 是第三轮访谈中客户硬约束要求入 MVP 的项（OQ-021 = R-Accept、OQ-022 = A、OQ-023 = S1、OQ-027 = A、OQ-028 = D2 + F3）。

**验收标准**：

- AC-010-1 平台提供一个用户级 Skill 库；同一作者名下可以拥有多个 Skill，名称按 [agentskills.io 规范](https://agentskills.io/specification) 唯一（1–64 字符，仅小写字母 / 数字 / 连字符，不以连字符开始 / 结尾，不连续连字符）。
- AC-010-2 用户能在线创建 / 编辑 Skill；UI 必须是**结构化表单**，将 frontmatter 拆为独立输入控件（`name` / `description` / `license` / `compatibility` / `metadata` 键值编辑器）+ Markdown body 编辑器；用户不会也不需要看到 YAML 语法（OQ-027 = A，NFR-001）。保存时由平台代为产出合规的 SKILL.md 文本。
- AC-010-3 用户能导入一个符合规范的 SKILL.md **单文件**；平台解析 frontmatter，校验 `name` / `description` 必填与格式；解析失败时拒绝并展示具体出错行 / 原因（详见 E7）。
- AC-010-4 导入遇到 `name` 与当前用户名下已有 Skill 重名时，提示用户选择"覆盖现有"或"另存为新名称"（OQ-028 = D2）。
- AC-010-5 Skill 与 Agent 的关联为**两层体系**：用户级 Skill 库存放原本；Agent 列表面上只存"已启用哪几个 Skill"的引用（OQ-028 = F3）。同一个 Skill 可被多个 Agent 引用。
- AC-010-6 用户能在 Agent 编辑页从自己的 Skill 库中勾选 / 取消勾选 Skill；保存后下一次试运行（REQ-004）必须反映该勾选集合。
- AC-010-7 试运行调用时，平台将已勾选 Skill 的 SKILL.md body 拼接进系统提示；拼接顺序 / token 预算控制由 H2 / H3 细化，本 REQ 不写死。
- AC-010-8 SKILL.md body 作为 system prompt 注入存在提示词劫持 / 绕过安全限制的风险；MVP 不做内容审核（OQ-012 = A），相关风险作为 12.1 节 R3 的拓展项登记。
- AC-010-9 MVP 阶段不提供以下能力（详见 ND-011、ND-012）：Skill ZIP / 多文件上传；`scripts/` 执行；Skill 商城 / 跨用户分享；Git URL 拉取。

**关闭依赖**：OQ-022 = A / OQ-023 = S1 / OQ-027 = A / OQ-028 = D2 + F3 均已 closed 2026-05-06。

### REQ-011 内置 Tool 启用 —— MVP

**说明**：用户能为自己的 Agent 启用平台提供的内置工具，Agent 在试运行时可调用。本 REQ 是第三轮访谈中客户硬约束要求入 MVP 的项（OQ-021 = R-Accept、OQ-024 = A、OQ-029 = T-1 + T-3）。

**验收标准**：

- AC-011-1 平台提供一份**固定**的内置工具清单；首批包含两项：
  - **T-1 联网搜索**：给定查询返回网页摘要 / 链接清单。
  - **T-3 当前日期 / 时间**：返回服务器当前时间戳（含时区）。
- AC-011-2 用户能在 Agent 编辑页**勾选 / 取消勾选**内置工具；不允许用户自定义工具（OQ-024 = A，与 NFR-001 一致）。
- AC-011-3 首次启用每个会外发数据的工具（首批中仅 T-1 联网搜索会外发）时，必须弹"用户须知"，明示对话内容会外发给工具服务方；未确认不启用（OQ-020 = B1，E6）。T-3 不产生外发，不需须知。
- AC-011-4 试运行中 Agent 调用工具产生的 token 与模型调用计入**同一**计费桶（OQ-020 = A1，NFR-005）；未启用的工具不会被调用。
- AC-011-5 工具调用失败 / 超时遵循与模型调用相同的重试策略（1 + 3，E1）；仍失败时以友好提示返回对话流，不影响后续轮次。
- AC-011-6 MVP 阶段不提供（详见 ND-013）：URL 抓取（T-4）、图片生成（T-5）、文件读取（T-6）、代码执行（T-7）、表格 / 图表（T-8）；用户也不能自定义工具。

**关闭依赖**：OQ-024 = A / OQ-029 = T-1 + T-3 均已 closed 2026-05-06。

### REQ-012 MCP server 集成 —— MVP UI 占位 + vNext 后端

**状态**：MVP 仅提供 UI 占位（“MCP 服务即将上线”），后端不接 MCP 协议；后端 vNext 阶段再实现（OQ-021 = R-Accept、OQ-025 = A）。

**验收标准**：

- AC-012-1 Agent 编辑页右侧实际提供三个 tab：Skill / Tool / **MCP**。MCP tab 在 MVP 中灰显（不可勾选），明示"即将上线"与一句能力简述。
- AC-012-2 MCP tab 不含任何可点击控件；不引用任何后端接口；不产生任何试运行动作。

## 6. 非功能需求

- NFR-001 终端用户为非技术用户，UI 不得直接暴露需要工程理解的概念（如 temperature、top_p、向量索引、MCP transport、JSON Schema、YAML frontmatter 语法）。MVP 阶段一律以"模板 / 向导"形态包装；Skill 编辑必须以结构化表单呈现（REQ-010 / AC-010-2）；vNext 阶段沿用相同约束（OQ-002 = A）。
- NFR-002 性能 / 可用性 / 并发：MVP 面向内部小范围使用，预期同时在线 ≤ 100 人；本特性内**不**承诺可用性 SLO，运行时可用性由平台运维模块覆盖（OQ-011 = A）。
- NFR-003 国际化 / 多端 / 可访问性：UI 支持简体中文 + 英文双语；覆盖 PC 浏览器与移动端响应式；可访问性按 WCAG 2.1 AA 等级实施（OQ-015 = B）。具体页面 / 状态矩阵在 H1 UI 说明阶段补齐。
- NFR-004 数据合规：平台**绝不**用用户在本特性内输入的 Prompt、知识库、Skill 内容、对话记录、Tool 调用参数 / 返回用于模型训练或评测；所有用户数据仅在用户所选地域驻留（OQ-008 = A）。
- NFR-005 计费口径：试运行中 Tool 调用（REQ-011）产生的 token 与模型调用产生的 token 计入**同一**计费桶，统一进入每用户每天调用额度（OQ-007 / OQ-020 = A1）；vNext 阶段接入 MCP 后端时，MCP 调用产生的 token 同样入同一桶。

## 7. 权限边界

- PB-001 一条 Agent 默认归创建者私有：创建者可读 / 可写 / 可删；其他用户不可见、不可调用、不可枚举。
- PB-002 任何"扩大可见域"的动作（如 vNext 的"公开"）必须由作者主动发起，不存在"默认对外可见"形态。
- PB-003 MVP 不引入"团队"角色与权限矩阵；vNext 也不引入团队（OQ-003 = A）。
- PB-004 vNext 阶段开放"公开"档时，可见域 = 平台已登录用户；**不**对互联网匿名开放；不存在匿名访客角色（OQ-004 = A）。
- PB-005 vNext 阶段分享 = 固定快照；作者改 / 删 Agent 不会改变已分享版本对使用者的呈现；版本号在使用者侧 UI 必须可见（OQ-009 = B）。

## 8. 数据边界

- DB-001 用户在 Agent 中输入的 Prompt、上传的知识库内容（vNext）、Skill 内容（REQ-010）、对话记录、Tool 调用参数 / 返回、Agent 元数据，归属创建者；MVP 不引入团队归属（PB-003）。
- DB-002 删除一个 Agent 必须不会留下"幽灵入口"（列表里没有了，但还能被原 URL 命中）。
- DB-003 平台**绝不**用用户内容用于模型训练 / 评测样本；用户数据仅在用户所选地域驻留（OQ-008 = A）。
- DB-004 删除 / 注销场景的归属处理：软删除并保留 7 天供用户自助恢复，过期后硬删除；账号注销时其名下所有 Agent 走相同 7 天软删除流程（OQ-013 = B）。
- DB-005 知识库内容（REQ-006 vNext / pending）的格式 / 单文件大小 / 累计大小上限：待 [OQ-019](./open-questions.md) 关闭后细化，本期不预定。
- DB-006 Skill 数据归属采用**双层模型**（OQ-028 = F3）：
  - 用户级 Skill 库存原本 SKILL.md 文本（包含 frontmatter 与 body）。
  - Agent 上只存 Skill 引用 ID + 当前版本号；Agent 不拷贝 Skill 内容。
  - Skill 作者修改 Skill 会立即影响所有引用该 Skill 的未分享 Agent（vNext：分享 Agent 锁定快照，PB-005）。
- DB-007 内置 Tool 启用配置（REQ-011）归属 Agent 级：Agent 记录已启用的工具 ID 集合与"首次启用须知是否已确认"标记（面向当前用户）；工具本身代码 / 密钥不在本特性内控制。
