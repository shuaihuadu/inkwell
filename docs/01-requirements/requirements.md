## 1. 项目背景

`inkwell` 旨在从 0 到 1 实现一个**面向团队 / 企业内部使用**的 LLM Agent 平台，让团队成员能在统一客户端中创建、配置、使用、共享自己或团队的 AI Agent。

项目当前的双重定位：

- **业务定位**：满足"团队成员（约 100 人量级）能自助创建 / 配置 / 使用 / 共享 LLM Agent"的内部生产力工具。
- **作者定位**：作者（项目负责人）"想体会从 0 到 1 实现一个 Agent 平台"的学习项目。

这两个定位在 v1 范围决策（见 [§5 功能范围](#5-功能范围) 与 [§9 上游决策与假设](#9-上游决策与假设)）上存在张力，已在第 9 节原文记录。

## 2. 目标用户

- 团队 / 企业内部成员（约 100 人量级）。
- 用户预期角色：**业务使用者**（创建并使用 Agent 解决日常问题）+ **Agent 配置者**（构建并维护 Agent 给自己或团队用）。两类角色由同一类账号承担，v1 不做权限拆分（详见 [§7 权限边界](#7-权限边界)）。

## 3. 用户角色

v1 仅一类主角色：**Member（已登录用户）**。

- 所有登录用户能力对等：均可创建 / 编辑 / 删除自己创建的 Agent，可使用其他成员共享出来的 Agent，可向团队共享自己创建的 Agent。
- v1 **不引入**编辑者 / 只读使用者等细粒度角色拆分；账号由后端运维人员通过 SQL / 管理脚本创建（依 [OQ-005 已关闭](./open-questions.md#oq-005-v1-账号开通方式管理员后台创建的具体形态)）。
- **Admin（Member 子集，`is_super=true`）**：依 [OQ-007 已关闭](./open-questions.md#oq-007-团队角色暂不区分的下游含义) v1 保留“最小管理员”能力。Admin 是在用户表上加一个标记位的 Member，平时能力与 Member 一致；额外可访问仅面向 Admin 的管理页（REQ-017），能力限于：解封账号、撤销其他成员的共享 Agent、查看全量审计日志。本属性不开放给用户自助升级，仅由后端运维在 SQL 中设置。
- 未来角色演进（细粒度角色 / 开放注册 / SSO）记录在 [§10 不做范围](#10-不做范围) 与 OQ-005 / OQ-007 决策纪录。

## 4. 核心场景

### 4.1 场景 S1：成员登录并查看 Agent 库

1. 成员打开 Electron 客户端，输入账号密码登录。
2. 客户端以**三档 tab** 展示 Agent 库（依 [OQ-010 已关闭](./open-questions.md#oq-010-v1-是否需要团队共享-agent-库独立于我的-agent)）：
   - **我的**：当前用户作为 Owner 创建的 Agent。
   - **团队共享**：其他成员共享出来、当前用户可只读使用的 Agent。
   - **我使用过**：当前用户**与之有过对话历史**的 Agent（默认判定依据；H3 可调整详细语义），可与“我的” / “团队共享” 的 Agent 重叠。
3. 列表项包含 Agent 头像、名称、描述、所有者、最近使用时间。
4. Admin（`is_super=true`）额外可从入口进入管理页（REQ-017）。

### 4.2 场景 S2：新建并配置 Agent

1. 成员在 Agent 库点击"新建 Agent"。
2. 系统创建一个 draft 状态的 Agent，进入配置页。
3. 成员依次配置：名称 / 头像 / 描述 / Instructions（System Prompt）/ 模型 / 模型参数 / 工具 / Skills / 知识库 / 长期记忆 / 触发器 / 多 Agent 编排 / 公开 API 配置。
4. 成员保存后进入"可使用"状态，并自动生成 v1 配置版本（见 §4.7 场景 S7 版本管理）。

### 4.3 场景 S3：与 Agent 对话

1. 成员选中一个 Agent，进入对话视图。
2. 成员可输入：**文本** / **图片** / **语音**（说话录入） / **文档**（拖拽上传）。
3. 客户端把消息发送至后端 API；后端把上下文（含 Skills、知识库召回、Function Calling）拼装后转发到选定模型。
4. Agent 回复以流式或一次性形式回到客户端展示，对话历史**全量持久化到后端**。

### 4.4 场景 S4：给 Agent 配置工具与 Skills

1. 成员在配置页打开"工具"面板，选择已注册的工具（Function Calling）勾选给 Agent；成员可填工具特定参数。
2. 成员在配置页打开"Skills"面板，从 Skill 库中选择若干 Skill 挂到 Agent；Skill 采用 [agentskills.io](https://agentskills.io/home) 的 `SKILL.md` 文件夹格式（仅 `SKILL.md` + `references/` + `assets/`，**v1 禁用** `scripts/`，详见 §10）。
3. 保存后，对话场景下后端按渐进加载（Discovery → Activation → Execution）让 Agent 在合适时机吸收 Skill 上下文。

### 4.5 场景 S5：调试与回放对话

1. 成员在 Agent 详情页打开"调试"视图。
2. 调试视图展示一次对话的全链路 trace：实际拼装的 prompt、Skills 命中情况、知识库召回片段、工具调用与返回、模型响应。
3. 成员可逐步骤回放，定位问题；每条调试记录可命名保存为评测样本。

### 4.6 场景 S6：多 Agent 编排与触发器

1. 成员在"编排"视图把多个 Agent 串成 DAG（Agent A 的输出作为 Agent B 的输入）。
2. 成员为编排设置触发器：定时（cron 表达式）或事件（HTTP webhook 入站）。
3. 编排被触发后由后端调度执行，结果可在客户端查看历史。

### 4.7 场景 S7：版本管理与回滚

1. 成员每次保存 Agent 配置时自动生成一个版本（`v1`、`v2` …）。
2. 成员可在"版本"视图查看版本列表、diff、回滚到任一历史版本。
3. 编排与公开 API 引用 Agent 时可锁定到具体版本（避免上游配置变更导致行为漂移）。

### 4.8 场景 S8：公开 API / Webhook 暴露

1. 成员在 Agent 详情页生成对外调用 Token（仅显示一次）。
2. 外部系统通过 HTTPS 调用 `inkwell` 后端 API，传入 Token + Agent 标识 + 输入，得到 Agent 输出。
3. 所有外部调用记入审计日志（见 §6 NFR-审计）。

>
> - **图片**：客户端可粘贴 / 拖拽图片，后端转发给支持视觉的多模态模型；不支持视觉的模型应在 UI 上拒收。

下列内容**不在产品 UI 内配置**，由后端配置文件管理（决策见 §9）：

- 各模型厂商的 API Key（Azure OpenAI / OpenAI / Claude / Qwen / 智谱 / …）。
- 模型路由策略（默认模型、降级顺序）。
- **ASR 服务凭证**（Azure Speech 密钥与区域，v1 默认接入；依 OQ-003）。
- 后端服务的端口、数据库连接、对象存储等基础设施配置。

## 6. 非功能需求（NFR）

> 项目负责人在 H1 阶段明确表示"非功能要求先不考虑，先把功能实现了"。本节因此**只列硬底线**（涉及合规、可追溯、跨平台、客户端会话），其余性能 / 可用性 / 并发指标按 [OQ-002 已关闭决策](./open-questions.md#oq-002-v1-是否给出最低性能--可用性底线) 在 v1 不设兜底，统一移入 [§10 不做范围](#10-不做范围)。

| ID      | 项             | 内容                                                                                                                                                                                                                                                                                       |
| ------- | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| NFR-001 | 联网要求       | 客户端与后端 API **强制联网**才能工作；v1 不支持离线模式（决策见 §9）。                                                                                                                                                                                                                    |
| NFR-002 | 客户端跨平台   | Electron 桌面端需同时支持 **Windows 11** 与 **macOS 12+（仅 Apple Silicon）**，两端功能完全等价；不要求 Linux，不支持 Windows 10 与 Intel Mac（依 [OQ-009 已关闭](./open-questions.md#oq-009-跨平台测试矩阵)）。                                                                           |
| NFR-003 | 客户端自动锁定 | 客户端**离开（前台失焦或无操作）累计 5 分钟**自动锁定，需要重新输入密码解锁。                                                                                                                                                                                                              |
| NFR-004 | 审计日志       | 系统必须保留可审计的操作日志，覆盖：登录 / 登出、Agent 创建 / 修改 / 删除 / 共享、Agent 调用（含来自公开 API 的调用）、版本回滚、Skill 与工具的挂载变更。日志可在 UI 中按用户 / Agent / 时间检索。                                                                                         |
| NFR-005 | 对话历史持久化 | 所有对话历史**全量存储到后端**，不依赖客户端本地保存；多端登录可看到一致历史。                                                                                                                                                                                                             |
| NFR-006 | 数据出境合规   | 依 [OQ-001 已关闭决策](./open-questions.md#oq-001-数据出境与合规边界)：v1 允许把对话内容、上传的图片 / 语音 / 文档原文发送给已签 DPA 的云端模型厂商；**v1 不做敏感字段脱敏、不做内容分类**；合规责任前置到厂商合同与人工审核。后端只需保留可审计的调用记录（见 NFR-004），不引入脱敏管线。 |

性能 / 可用性 / 并发等指标在 v1 不做硬性约束（[OQ-002 已关闭](./open-questions.md#oq-002-v1-是否给出最低性能--可用性底线)，决策选项 B：所有性能问题归类为"待优化"，不算缺陷）。下列**软目标**仅作为 H2 / H3 设计输入，不作为缺陷判定依据：

- 用户量级：~100。
- 单用户 Agent 数：默认不限，但提供配置项可设上限。
- 单 Agent 知识库总大小：默认不限，但提供配置项可设上限。
- 单 Agent 单次对话最大轮数：默认不限，但提供配置项可设上限。

> **注**：EX-007（编排死循环 / 超时强制终止）属于**安全 / 正确性兜底**，不是性能 SLA，不受 OQ-002 决策影响——v1 仍必须有 max-steps / max-duration 上限，避免编排自爆。具体上限值由 H3 决定。

## 7. 权限边界

- **登录态**：所有用户操作（除登录本身）必须基于有效登录态。
- **角色**：v1 仅 Member 主角色 + Admin 子集（见 §3）。
- **资源所有权**：每个 Agent 属于一个 Owner（创建者）；Owner 可编辑 / 删除 / 共享 Agent。共享后的 Agent 对其他成员**只读使用**（不能编辑或删除）；其他成员可“复制为我的 Agent”获得一份新 Owner 副本。
- **Admin 越权能力**（仅 `is_super=true` 的 Member）：可解封账号、可撤销他人的共享 Agent（仅取消共享可见性，不删除 Owner 原件）、可查看全量审计日志；不能编辑他人的 Agent 配置。依 [OQ-007 已关闭](./open-questions.md#oq-007-团队角色暂不区分的下游含义)。
- **对外调用**：公开 API Token 与具体 Agent 绑定；Token 只能调用绑定的 Agent，不能跨 Agent 使用；**一个 Agent 同时只能存在 1 个有效 Token，新建即作废旧的**（依 [OQ-004 已关闭](./open-questions.md#oq-004-公开-api-token-的撤销与轮换策略)）。
- **客户端会话**：5 分钟无操作锁定（见 NFR-003）。

## 8. 数据边界

### 8.1 涉及的数据类别

- **账号数据**：用户名、密码哈希、登录历史、最近一次活动时间。
- **Agent 配置数据**：基础属性、Instructions、模型选择 / 参数、工具挂载、Skill 挂载、知识库引用、记忆策略、触发器配置、编排配置、公开 API Token 元数据、版本树。
- **运行期数据**：对话消息（文本 / 图片 / 语音转写文本 / 文档摘录）、工具调用记录、模型调用日志、调试 trace、评测样本、定时 / 事件触发的执行记录。
- **资产数据**：用户上传的图片、语音、文档；Skill 携带的 `references/` 与 `assets/` 文件。
- **审计数据**：见 NFR-004。

### 8.2 数据存储策略

- **对话历史**：v1 强制全部存到**后端**（用户跨设备同步 + 满足审计 / 合规要求）。
- **Agent 配置 / 资产 / 审计 / 运行期数据**：均由后端持有，客户端按需获取。
- **API Key（模型厂商）**：仅存在于后端配置文件，**不下发到客户端**。

### 8.3 数据生命周期

v1 默认策略（具体保留期由后端配置项控制，初始默认值由 H2 / H3 决定）：

- 账号：永久保留。
- Agent 配置：永久保留；**删除为硬删除，不支持恢复**（详见 [HD-002 §1.3 Q5](../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md)）。
- 对话历史：永久保留，可由 Owner 删除。
- 审计日志：至少保留 6 个月（v1 默认值，可配置）。
- 临时上传文件（解析后已入向量库）：原文件保留以便复核 / 重新解析。

> 数据出境合规由 [NFR-006](#6-非功能需求nfr) 统一规约：v1 允许原文外发至已签 DPA 的云端模型厂商，不做脱敏 / 不做内容分类（[OQ-001 closed 2026-05-07](./open-questions.md#oq-001-数据出境与合规边界)）。

### 8.4 数据所有权

- Agent 创建者拥有 Agent 配置数据所有权。
- 用户与 Agent 的对话历史归对应**用户**所有；分享 / 共享 Agent 不会带走对方的历史对话。
- 公开 API 调用产生的数据归对应 Agent 的 Owner 所有。

## 9. 异常场景

| 编号   | 场景                                             | 期望行为（H1 边界，UI 细节交 H1-UISpecAuthor）                                                   |
| ------ | ------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| EX-001 | 后端 API 不可达                                  | 客户端**禁止**降级到本地缓存对话，应明确告知用户"网络异常，已断开"，禁止继续提交消息。           |
| EX-002 | 模型厂商不可用 / 限流 / 超时                     | 后端按配置的降级策略处理；用户层得到结构化错误（含错误码 + 重试建议），不得静默失败。            |
| EX-003 | 工具调用失败                                     | 把工具失败原因显式注入对话上下文；Agent 可选择继续推理或终止本轮。                               |
| EX-004 | 多模态输入与所选模型不兼容（如发图给纯文本模型） | UI 拒收并提示"当前模型不支持图像输入"；不得提交后失败。                                          |
| EX-005 | 公开 API Token 鉴权失败 / 速率超限               | 返回标准 HTTP 4xx + 错误码；记入审计日志；不暴露内部细节。                                       |
| EX-006 | 客户端 5 分钟空闲                                | 自动锁定，需重新输入密码解锁；锁定期间网络层会话保留，重输密码后无需重登。                       |
| EX-007 | 编排死循环 / 超时                                | 后端有最大执行步数与最长执行时间（默认值由 H3 决定，但属于 v1 范围），超出后强制终止并记入审计。 |
| EX-008 | Skill 内容缺失 / 损坏                            | Agent 加载 Skill 失败时不阻断对话，但要给出明确警告，避免静默"Skill 没加上"。                    |
| EX-009 | 文件上传超出大小 / 格式不支持                    | 客户端先拦截；后端二次校验。具体限值在 H3 决定。                                                 |

## 10. 不做范围（v1 明确不做）

- **不做**：Skill 携带 `scripts/` 的执行（仅支持 `SKILL.md` + `references/` + `assets/`）。
- **不做**：本地大模型接入（v1 不接 Ollama / vLLM / llama.cpp）。
- **不做**：Skill 市场 / 社区分享生态（Skill 由后端管理员或用户上传到自家库，不公开发布）。
- **不做**：跨团队 / 多组织（v1 一个部署一个团队）。
- **不做**：计费 / 用量统计 / 配额硬限制（仅有"上限可配置"的软限制）。
- **不做**：移动端 / Web 端（仅 Electron 桌面）。
- **不做**：Linux 桌面、**Windows 10**、**Intel Mac** 与 **macOS 11 及以下**（依 [OQ-009](./open-questions.md#oq-009-跨平台测试矩阵) 仅保 Win11 + macOS 12+ Apple Silicon）。
- **不做**：用户自助注册 / SSO / 公网 OAuth（v1 由后端管理员通过 SQL / 管理脚本创建账号，依 [OQ-005](./open-questions.md#oq-005-v1-账号开通方式管理员后台创建的具体形态)）。
- **不做**：细粒度角色拆分（编辑者 / 只读使用者等）；v1 仅保留 Member + Admin（`is_super`）两档，依 [OQ-007](./open-questions.md#oq-007-团队角色暂不区分的下游含义)。
- **不做**：公开 API Token 的多 Token 并存 / 过期轮换策略；v1 仅支持“单 Token，新建即作废”（依 [OQ-004](./open-questions.md#oq-004-公开-api-token-的撤销与轮换策略)）。
- **不做**：可配置的数据保留期 UI（保留期通过后端配置文件改，不暴露到产品 UI）。
- **不做**：v1 阶段对性能 / 可用性 / 并发的 SLA 承诺（移入 v2 视情况补）。
- **不做**：离线模式（任何离线场景；客户端必须连接后端 API）。

## 13. 上游决策与假设

> 本节原文记录访谈中关键回答，供后续阶段追溯。复述、总结、推论一律不放本节。

1. **Agent 类型**：用户原话——"基于大语言模型的Agent"。
2. **目标用户**：用户原话——"团队/企业内部使用"。
3. **交付形态**：用户原话——"单机桌面应用的方式，就是Electron应用，后台是API"。
4. **项目目的**：用户原话——"想体会从 0 到 1 实现一个 Agent 平台"。
5. **MVP 边界**：用户原话——"先从小的入手，M3 单机客户端 + 远端共享 API"。
6. **配置维度**（v1 全勾）：用户原话——"名称 / 头像 / 描述、System Prompt / 人设、选择底层模型……、调试 / 评测、版本管理"+"用户故事：用户登录登录，然后显示已经配置的Agent列表，新建Agent可以设置名称、头像、Instructions、选择不同的模型、保存，保存之后可以直接对话和使用，可以支持图片、语音、文档多模态输入。用户还可以给Agent设置工具、Agent Skills"。
7. **范围分层**（被告知工作量风险后）：用户原话——"我坚持 v1 全做 16 项"。**风险记录**：用户在被明确告知工作量与个人学习项目体量不匹配后，仍坚持 v1 全做。后续若 H5 阶段无法完成，应回到 H1 重新分层，而非降低质量交付。对应 OQ-006。
8. **角色权限**：用户原话——"暂时只有用户，先不设置权限"。
9. **模型接入**：用户原话——"一开始就要支持多家云服务，但是先支持Azure OpenAI（OpenAI / Claude / Qwen / 智谱 / ...），这些配置在服务器端端配置文件"。
10. **离线**：用户原话——"客户端应用肯定得联网啊，不联网怎么访问服务器，所以永远只联网"。
11. **Skill 沙箱边界**：用户原话——"A. v1 的 Skill 不允许带 scripts，只支持 SKILL.md + references/ + assets/（纯指令 + 参考材料 + 模板）"。
12. **数据存储 / 量级 / 对话归属**：用户原话——"数据存储由后台决定 / v1用户量可以考虑100 / 单用户Agent数量默认不限、知识库默认不限、对话最大轮数默认不限，但是可以配置 / 对话历史全存到后端，满足合规要求"。
13. **NFR**：用户原话——"非功能性要求先不考虑吧，先把功能实现了"。
14. **审计日志 / 客户端锁定**：用户原话——"审计日志是需要的，客户端离开超时5分钟自动锁定"。
15. **账号体系**：用户原话——"账号体系先使用最简单的，但是未来需要开放注册、接入其他SSO和互联网公开的OAuth"。
16. **Electron 跨平台**：用户原话——"Electron客户端需要跨平台，Windows+MacOS"。
17. **数据出境合规边界**（OQ-001 closed 2026-05-07）：用户原话——"公司已与所有候选模型厂商签 DPA，对话与文件可原文发送，不做脱敏。后果：实现成本最低；合规责任全部前移到厂商合同与人工审核。" 已落到 NFR-006 与 §8.3。
18. **v1 性能 / 可用性兜底**（OQ-002 closed 2026-05-07）：用户原话——"不同意，v1 完全不设兜底，所有性能问题先归类为'待优化'，不算缺陷。" **风险记录**：用户已签字接受 OQ-002 卡片明示的下游后果——"H4 无法形成性能用例；触发器 / 编排可能跑死无人察觉"。EX-007 安全兜底不受影响，由 H3 设上限值。
19. **多模态语音 ASR 路由**（OQ-003 closed 2026-05-08）：用户原话——"后端独立 ASR（v1 优先 Azure Speech，与 Azure OpenAI 同生态）；语音先转文本再注入对话，所有模型都能用。后果：开发简单；丢失语音原始信息（情绪 / 重音）。"
20. **公开 API Token 策略**（OQ-004 closed 2026-05-08）：用户原话——"一个 Agent 只能有 1 个有效 Token；新建即作废旧的。后果：UI 简单；调用方切换有窗口期。"
21. **账号体系形态**（OQ-005 closed 2026-05-08）：用户原话——"v1 用户名 + 密码；账号由后端管理员通过 SQL / 管理脚本创建；不引入抽象层（v2 再重构）。后果：v1 最快；v2 引入 SSO 时要重做用户表与登录链路。"
22. **v1 范围风险接受签字**（OQ-006 closed 2026-05-08）：用户原话——"我承认风险，接受为已知风险继续推进。若 H5 阶段无法完成，回 H1 重新分层。" 本条与第 7 条合并作为风险转移凭证；后续阶段不为体量问题打回范围决策。
23. **Admin / 超级成员**（OQ-007 closed 2026-05-08）：用户原话——"v1 直接做出最小管理员页（解封账号、撤销共享、查全量审计），仅向 is_super=true 的成员开放。"
24. **长期记忆语义**（OQ-008 closed 2026-05-08）：用户原话——"选 C。该设计点推迟到 H3，由详细设计中的数据表设计决定。" 本 H1 仅保留 “开 / 关 / 摘要” UI 期望，后端与对话表的归属关系由 H3 决定后回填。
25. **跨平台测试矩阵**（OQ-009 closed 2026-05-08）：用户原话——"Windows 11 + macOS 12+（仅 Apple Silicon）；功能完全等价。后果：矩阵 2 项；放弃 Intel mac 与老 Win10。"
26. **Agent 列表资产架构**（OQ-010 closed 2026-05-08）：用户原话——"三档 tab：'我的' / '团队共享' / '我使用过'；以发现性为优先。后果：信息架构最复杂，需 H3 跟踪'使用过'语义（默认以'有过对话'为判定）。"
