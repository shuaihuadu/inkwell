# ADR-022 Entity ↔ Model Mapper 选型：手写扩展方法（`Entity.ToModel()` / `Model.ToEntity()`）

## 上下文

[ADR-021](./ADR-021-efcore-persistence-shared-base-and-provider-csproj-layout.md) 锁定 EFCore Persistence 4-csproj 布局——业务 Entity 集中于 `providers/Persistence/Inkwell.Persistence.EFCore/Entities/`；[ADR-017 §3.2](./ADR-017-backend-module-topology-ports-and-adapters.md) + [HD-002 Q1=C facade only](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 锁定业务命名空间不能直接 `using Microsoft.EntityFrameworkCore.*`、不能查询 Entity 类——业务命名空间持有的是**业务 Model**（定义于 `Inkwell.Abstractions/Persistence/<Module>/`），具名 `IXxxRepository` 接口签名也是 Model 维度。

由此引出**强制存在的 Entity ↔ Model 映射边界**：写入路径 Model → Entity 入库；读取路径 Entity → Model 出 facade；List 路径 `IQueryable<Entity>` → 投影成 Model（理想下投影下推到 SQL）。

H3 起草 [HD-002 §11 待补](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 中明确 mapper 选型未锁；[`design-review-report.md` §6.4.1 F6](../../04-detailed-design/design-review-report.md) Owner 反馈要求起 ADR-022 锁选型——避免 mapper 选型在 HD-009 内部反复改写、避免业务 HD 起草具名 Repository 时各自挑工具。

驱动因素：

- **零商务依赖原则**：v1 是 dogfood 项目（[`AGENTS.md` §1](../../../AGENTS.md)），无商务采购预算——任何带商业 license 的库直接出局。
- **AOT / trim 友好**：[ADR-002](./ADR-002-backend-runtime-dotnet10-aspnetcore.md) 选 .NET 10 + ASP.NET Core 10——v1 不强求 AOT 发布，但任何阻碍未来 AOT 路径的依赖都是负担（[Native AOT 反射限制](https://learn.microsoft.com/dotnet/core/deploying/native-aot/limitations)）。
- **可观测性义务**：[ADR-013](./ADR-013-observability-otel-self-hosted-grafana.md) 要求所有错误带 `error.code` 与 `trace.id`——映射层异常（如未来字段漏填、类型转换失败）必须在源码里能直接定位到行，不能跳进生成代码 / 反射栈。
- **mixin 体系**：[HD-002 §3.7 ~ §3.9](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 三 mixin（`IHasTimestamps` / `IHasRowVersion` / `IHasOwner`）= Entity 与 Model 都按需实现 → 字段对齐天然存在；mapper 不需做"改名 / 类型转换"魔法。

## 决策

**采用手写 mapper，按 Extensions 模式组织——`Entity.ToModel()` / `Model.ToEntity()` / `IQueryable<Entity>.SelectAsModel()` 三方法静态扩展，per-entity 一文件。**

### 核心边界（4 条）

1. **业务接口只见 Model，不见 Entity**——`Entity` 类型与 EFCore SDK 限定于 `providers/Persistence/Inkwell.Persistence.EFCore/` 内部，业务命名空间不导出 / 不引用。
2. **必须三向**——`ToModel` / `ToEntity` / `SelectAsModel`，对每个 Entity 都齐备；`SelectAsModel` 在 List / Pagination 路径下推为 SQL 列投影（`Select(e => e.ToModel())` 不可被 EFCore 翻译，[client-eval 限制](https://learn.microsoft.com/ef/core/querying/client-eval)）。
3. **禁隐式转换**——字段名不一致必须在 mapper 里显式书写；不靠"约定推断"——这是手写相对 Mapperly / AutoMapper 的核心差异。
4. **禁引入外部映射库**（AutoMapper / Mapperly / Mapster 等）——license、AOT、trace 三维取舍均不利（详 §备选项）。

> 命名规则（Model 默认无后缀 / 撞名降级 `XxxDefinition` / Repository 动词白名单 `Add` / `Update` / `Get` / `Delete` / `List` / `Find` / 禁 `XxxModel` `XxxDomain` `XxxDTO` 后缀）、`Mapping/` 子目录、空守护 / mixin 字段处理 / `AuditingSaveChangesInterceptor` 联动、测试覆盖率门槛（≥ 95%）、CI grep 自检命令均下沉到下游设计——见 §迁移路径。

## 备选项

### 备选 A（被选用）：手写 mapper 扩展方法

- **被选用**：
  1. **零依赖、零 license 风险**：不引入任何映射库——映射代码即业务代码，受 Inkwell 自有 license 与 CI 完全控制；与 [`AGENTS.md` §3.3 "v1 不引入外部中间件"](../../../AGENTS.md) 风格一致。
  2. **AOT / trim 100% 友好**：纯静态方法 + 显式属性赋值，零反射、零运行期 IL 生成；与 [.NET 10 Native AOT 路线](https://learn.microsoft.com/dotnet/core/deploying/native-aot/) 完全相容（v1 即便不发 AOT，也不留未来阻塞）。
  3. **trace 可定位性最强**：映射层任何字段错误都能直接打开源文件 + 行号定位，无需跳进 source generator 产物或反射栈——与 [ADR-013 OTel 全链路追溯](./ADR-013-observability-otel-self-hosted-grafana.md) 与 [REQ-014 trace 完整性](../../01-requirements/requirements.md) 取向一致。
  4. **显式 > 隐式**符合 .NET 10 / C# 14 的整体语感（[`<Nullable>enable</Nullable>`](https://learn.microsoft.com/dotnet/csharp/nullable-references) / `init` 属性 / `required` 关键字均在向"显式声明"演进）；mixin 字段在 Model / Entity 同名直传，mapper 没有"魔法"消化空间——boilerplate 代价可接受。
  5. **`SelectAsModel` 性能可控**：手写 `Select(entity => new Model(...))` 是 EFCore 最容易识别 + 翻译成 SQL `SELECT col1, col2, ...` 的写法，等价于 AutoMapper `ProjectTo<T>()` 但无中间层；List 查询路径不被映射方式拖慢。

### 备选 B：[Mapperly](https://github.com/riok/mapperly) v3+（source generator）

- **理由**：源生成 = 编译期错、零反射、AOT 友好；MIT license 免费；活跃社区（[`riok/mapperly` Stars / Releases](https://github.com/riok/mapperly/releases)）；2025 年已稳定 v3+。
- **放弃理由**：
  1. **trace 可定位性退化**：错误栈跳进生成代码（`<auto-generated>` partial）—— Owner 在评审中明确表态"显式可读、源码即真相"，与生成代码模式取向冲突；HD-002 Q1=C facade only 拓扑下，调试 Repository 内部 mapping 是高频活动，手写胜过生成。
  2. **mixin 字段一致性靠工具链**：Entity 与 Model 都实现 `IHasTimestamps` / `IHasRowVersion` / `IHasOwner`，字段同名同类型直传——Mapperly 的 convention-based mapping 在这里几乎不创造价值；"声明式 partial class + `[Mapper]` attribute" 与"显式 `new Model(...)` 一行"代码量差距小，引入额外构建期源生成器不划算。
  3. **未来字段重命名风险**：Mapperly 会在源生成期"自动桥接"配置成相似的字段——本仓库希望显式 fail（Owner 在 [HD-002 §4 命名约束](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 已立"不靠约定推断"立场），手写更安全。
  4. **零依赖原则的纯净度**：providers/Persistence/Inkwell.Persistence.EFCore base 已经引 EFCore SDK 是必要恶；再引 source generator 会让 csproj 编译图多一节点、CI 增加 build 时间——与 [ADR-021](./ADR-021-efcore-persistence-shared-base-and-provider-csproj-layout.md) "providers/* 仅引该 Provider 自身 SDK" 风格冲突。

### 备选 C：[AutoMapper](https://docs.automapper.org/) v15+

- **理由**：业界最知名 .NET 映射库；`ProjectTo<T>()` 投影 IQueryable 至 Model 已成熟；插件生态丰富。
- **放弃理由**：
  1. **license 死穴**：[Jimmy Bogard 2024.08 公开 AutoMapper / MediatR 转商业 license](https://www.jimmybogard.com/automapper-and-mediatr-going-commercial/)——v15+ 商用需付费，与 v1 dogfood 项目零商务预算冲突；这一条已是充分否决理由，不再展开讨论其他维度。
  2. **AOT 不友好**：AutoMapper 大量依赖反射 + 表达式树 IL emit，[Native AOT 路线](https://learn.microsoft.com/dotnet/core/deploying/native-aot/limitations) 不支持。
  3. **trace 可定位性最弱**：错误栈在反射桥 + 表达式编译缓存里，远比 Mapperly 还难追踪——与 [REQ-014 trace 完整性](../../01-requirements/requirements.md) 取向反向。
  4. **convention magic**：AutoMapper 默认按属性名约定推断映射，与 Owner "显式 > 隐式" 立场冲突。

### 备选 D：手写 mapper 但不分 Extensions、集中放进 Repository 内私有方法

- **理由**：mapper 完全私有给 Repository，不暴露 internal 静态扩展，封装更严。
- **放弃理由**：
  1. 重复性最高：`AgentRepository` 的 `AddAgent` / `UpdateAgent` / `GetAgent` / `ListAgents` / `FindAgentsByOwner` 等方法都需调 `MapToModel` / `MapToEntity`，私有方法路径每个 Repository 重写一份；扩展方法路径全 Repositories 共享一份；
  2. List 路径的 `IQueryable<TEntity>.SelectAsModel()` 投影下推**必须**是扩展方法（`Select(e => e.ToModel())` 不可被 EFCore 翻译成 SQL，[`Select` lambda 内调实例方法 EFCore 拒绝翻译](https://learn.microsoft.com/ef/core/querying/client-eval)）—— 私有方法路径根本走不通；
  3. 测试入口繁琐：扩展方法可独立 ut 直接调；私有方法必须经 Repository 测试，测试粒度被迫粗化。

## 后果

### 正面

- **零外部依赖**：mapper 即仓库自有代码，license / AOT / trim / trace 全部内部可控；
- **`SelectAsModel` 投影可下推**：List / Pagination 查询路径仅 `SELECT` 必要列，不物化整行——性能等价 AutoMapper `ProjectTo<T>()` 但无库；
- **trace 直达源行**：所有映射错误（字段漏填 / 类型不匹配编译期已挡 / null 入参 ArgumentNullException）都能在源码定位，与 [ADR-013](./ADR-013-observability-otel-self-hosted-grafana.md) `error.code` + `trace.id` 全链路追溯一致；
- **mixin 字段对齐天然**：三 mixin 在 Entity 与 Model 同名同类型直传，mapper 不需做改名魔法——`new Model(...)` 一行覆盖；
- **业务 Model 对外零后缀可读性高**（[Owner 第二轮第 4 条](#上下文)）：业务代码中 `Agent` / `Conversation` / `Skill` 不带装饰后缀，阅读接近领域语言；仅撞名场景才加 `XxxDefinition`，语义准确；
- **per-entity 一文件** + **per-Repo 动词集**（[Owner 第二轮第 3 条](#上下文)）便于 PR diff、git blame、新 contributor 学习曲线低（不需学映射 DSL，也不需猜查询动词、选 `Find` / `Query` / `Fetch` / `Retrieve`）。

### 负面

- **boilerplate 代价**：每个 Entity × 3 方法（`ToModel` / `ToEntity` / `SelectAsModel`）+ 对应测试文件——比 Mapperly partial class + `[Mapper]` attribute 模板代码量翻倍；
- **mixin 体系演进风险**：未来若加新 mixin（如 `IHasTenantId` 多租户）——每个 mapper 文件需逐一 +字段 +测试，以及一个查找器扫漏（PR review 容易漏）；缓解方案：起 H5 编码任务 batch 改 + Roslyn analyzer 强制扫"未实现新 mixin"的 Entity；占位 [RISK-018](../risk-analysis.md)；
- **跨字段 / 嵌套 Model 映射成本**：未来若 Model 含嵌套 value object（如 `Agent.Settings : AgentSettings`），手写嵌套映射成本递增——v1 范围内**显式禁止** Model 嵌套（[HD-002 §4.1](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 锁 Model 为扁平 record）；若 v2 / v3 需嵌套，回炉本 ADR；
- **List 路径 `SelectAsModel` 不能复用 `ToModel`**：因 EFCore client-eval 限制，`SelectAsModel` 必须 inline 写 `new Model(entity.Field1, entity.Field2, ...)`，与 `ToModel` 双重维护——靠测试断言两者结果一致避免漂移（详 HD-009 §8 测试要求）。

### 中性

- **mapper 位置**——`providers/Persistence/Inkwell.Persistence.EFCore/Mapping/` 子目录（EFCore base csproj 同时见 Entity 与 Model via ProjectReference）；不放进 `Inkwell.Core` 因业务命名空间不应见 Entity。
- **csproj 数不变**——13（[ADR-021](./ADR-021-efcore-persistence-shared-base-and-provider-csproj-layout.md)）→ 13；mapper 仅是 EFCore base csproj 内部新增子目录。
- **新 mixin 扫漏机制**——本 ADR 不强制 Roslyn analyzer，但鼓励 HD-009 起草时一并落 `MissingMixinFieldAnalyzer`（扫描 Entity 实现某 mixin 但 mapping 文件未提及对应字段时编译报警）；进 [RISK-018](../risk-analysis.md) 占位。

## 置信度

`high`。

依据：(1) 手写 + 扩展方法是 .NET 生态成熟模式（参考 [`Microsoft.AspNetCore.Identity` `IdentityResult` extensions](https://learn.microsoft.com/aspnet/core/security/authentication/identity) / [`MediatR` 商业化前的 `IRequestHandler<,>` 直拼模式](https://github.com/jbogard/MediatR) / [Marten 的 record-style projection](https://martendb.io/) 等众多 .NET 生产代码库），无方法论风险；(2) AutoMapper 商业化是公开事实（[Jimmy Bogard 博客 2024.08](https://www.jimmybogard.com/automapper-and-mediatr-going-commercial/)），可机械验证；(3) Mapperly 的 source-gen 模式与本仓库"显式 > 隐式"立场的取向冲突可在源码层面机械识别（[Owner 在 HD-002 §4 命名约束](../../04-detailed-design/Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md) 已立明文）；(4) `SelectAsModel` 投影下推为 SQL 是 EFCore 文档明列模式（[Querying / Projection](https://learn.microsoft.com/ef/core/querying/)），不依赖外部库；(5) Owner 第二轮反馈的 4 条新规都能靠 [Roslyn analyzer / `BannedSymbols.txt`](https://learn.microsoft.com/dotnet/fundamentals/code-analysis/quality-rules/) 在 CI 中机械检验，不靠 reviewer 主观判断。
