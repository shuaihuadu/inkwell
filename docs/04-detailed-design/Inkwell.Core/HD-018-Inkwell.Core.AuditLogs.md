## 0. 范围核实：HD-007 查询侧现状 + 本 HD 实际边界

**核查结论（与任务描述的初始假设不完全一致，如实记录）**：

1. 已 reviewed 的 [HD-007 `IAuditLogger`](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#31-auditiauditloggercs) **已经**在同一个 facade 接口里声明了查询能力——`Task<PagedResult<AuditLogEntry>> QueryAsync(AuditLogQuery query, CancellationToken ct = default)`；查询过滤条件 `AuditLogQuery`（[HD-007 §3.4](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#34-auditauditlogquerycs)，含 `TimeRange` + `Pagination` + 可选 `ActorUserId`/`EventType`/`AgentId`）与查询结果 `AuditLogEntry`（[HD-007 §3.3](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#33-auditauditlogentrycs)）均已完整定义。**不存在**独立的 `IAuditLogReader` 之类的查询专用接口——查询能力就是 `IAuditLogger.QueryAsync` 本身，本 HD 不新增、不重复定义任何查询接口。
2. [HD-007 §1.3 Q-implementation-topology](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#13-关键决策摘要) + [file-structure.md `## Inkwell.Abstractions.Audit`](../file-structure.md#inkwellabstractionsaudit) 已明确锁定：`IAuditLogger` **只有一个**具体实现类——`Inkwell.Core.AuditLogs.DefaultAuditLogger`，file-structure.md 甚至已经预先写死了该类的文件名（`DefaultAuditLogger.cs` + `AuditLoggerBuilderExtensions.cs`）。这意味着**写入路径（`LogAsync` 的内部重试队列 / 磁盘 fallback）与查询路径（`QueryAsync`）必须实现在同一个类里**——不存在"只实现查询侧、写入侧留给未来另一个 HD"的拆分空间，因为 DI 容器只会注册一个 `IAuditLogger` 实现，两个方法体都要有真实代码。
3. 因此，任务描述"本 HD 只做查询侧...不改动 `IAuditLogger` 本身"这句话里，"不改动 `IAuditLogger` 本身"（即不碰接口签名 / 不碰 HD-007 正文）是完全成立且本 HD 严格遵守的；但"只做查询侧"如果理解为"不需要写`LogAsync`的具体实现"，与上述第 2 点锁定的单实现类拓扑不兼容。本 HD 的实际处理方式是：**完整实现 `DefaultAuditLogger`（`LogAsync` + `QueryAsync` 两个方法均有真实代码），但写入侧的全部技术细节严格照抄 [ADR-008](../../03-architecture/adr/ADR-008-audit-log-store-and-query.md) + [HD-007 §1.4](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#14-logasync-异步解耦声明呼应-adr-008业务事务不阻塞设计意图) 已经锁定的契约（内存队列 + 重试 3 次 + 磁盘 fallback + 告警），不做任何超出这些既有决策的新设计**；本 HD 真正投入详细设计精力的是查询侧（`QueryAsync` 的分页/过滤实现、`IAuditLogRepository` 查询方法、`AuditLog` 持久化 Model、UI-009 检索表单字段映射）。此项差异已列入 [§8 Q&A-A](#8-需要-owner-确认的问题) 供 Owner 确认是否符合预期，**不代为裁定"是否应该拆成两个 HD"这个流程性问题**。
4. `AuditLog` 持久化 Model + `IAuditLogRepository`（[HD-007 §1.2 "不在内"](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#12-范围) 已明确指派给本 HD）是**全新类型**，HD-007 未定义过，本 HD 在 `Persistence/AuditLogs/` 新增，不与 HD-007 的 `Audit/AuditLogEntry.cs`（查询结果投影）混淆——`AuditLogEntry` 是端口层查询结果 DTO，`AuditLog` 是本 HD 的持久化业务 Model，两者字段基本对齐但类型不同，`DefaultAuditLogger.QueryAsync` 内部做 `AuditLog → AuditLogEntry` 投影映射（详见 [§3.5](#35-inkwellcoreauditlogsdefaultauditloggercs)）。

**依赖规则遵循**（[AGENTS.md §3.2](../../../AGENTS.md)）：`Inkwell.Core.AuditLogs` 只依赖 `Inkwell.Abstractions` + BCL（含 `System.Threading.Channels` / `System.Text.Json`）；**不** `using` 任何 Provider 包，**不** `using Microsoft.Agents.AI.*`；持久化经 `IPersistenceProvider.GetRepository<IAuditLogRepository>()`（事务外读，`QueryAsync`）/ `IUnitOfWork.GetRepository<IAuditLogRepository>()`（事务内写，[HD-002 §13.3 Q1=A2](../Inkwell.Abstractions/HD-002-Inkwell.Abstractions-persistence-port.md)）。

**关于"Persistence 层 Options 新文件放置位置"的说明（非臆造 HD-007 未声明的字段，而是新增独立文件）**：[HD-007 §11 待补/待评审](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md) 明确"磁盘 fallback 文件的具体格式 / 路径 / 恢复流程"与"后台清理任务的调度实现"均**留给本 HD 独立锁定**，且"**严禁**回填到 `Inkwell.Abstractions/Audit/AuditLoggerOptions.cs`"（file-structure.md 同款措辞）。本 HD 遵循这一约束：**不修改** HD-007 已有的 `AuditLoggerOptions.cs` / `AuditLoggerOptionsValidator.cs` 两个文件一个字符，而是在同一 `Inkwell.Abstractions/Audit/` 目录下新增两个**独立文件** `AuditLogWriterOptions.cs` + `AuditLogWriterOptionsValidator.cs`（详见 [§3.3](#33-auditauditlogwriteroptionscs) / [§3.4](#34-auditauditlogwriteroptionsvalidatorcs)），与 HD-007 原有 7 个文件互不冲突、互不覆盖——这是延续 HD-007 自身"该目录预留给本 HD 追加"的既定安排，不是本 HD 越权修改 HD-007 正文。

## 1. 模块概述

### 1.1 职责

`Inkwell.Core.AuditLogs` 承担：

- **`IAuditLogger` 唯一实现**（`DefaultAuditLogger`）：`LogAsync`——把 `AuditLogRequest` 映射为 `AuditLog` 后写入进程内 `Channel<AuditLog>`（[HD-007 §1.4](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#14-logasync-异步解耦声明呼应-adr-008业务事务不阻塞设计意图) 已声明的异步解耦，`Task` 完成即代表"已入队"，不等待实际持久化）；`QueryAsync`——校验 `AuditLoggerOptions.MaxQueryTimeRangeDays`/`MaxPageSize` 后经 `IAuditLogRepository.ListAuditLogs` 分页查询，将 `AuditLog` 投影为 `AuditLogEntry` 返回（[NFR-004](../../01-requirements/requirements.md)"日志可在 UI 中按用户 / Agent / 时间检索"）
- **后台写入管道**（`AuditLogWriteBackgroundService`）：消费 `Channel<AuditLog>`，按 [ADR-008](../../03-architecture/adr/ADR-008-audit-log-store-and-query.md) 已锁定策略重试 3 次持久化，3 次均失败后经 `AuditLogFallbackFileWriter` 写磁盘 fallback 文件并触发告警指标
- **保留期清理**（`AuditLogRetentionCleanupBackgroundService`）：按 `AuditLoggerOptions.RetentionDays`（默认 180 天）周期性删除过期记录（[ADR-008](../../03-architecture/adr/ADR-008-audit-log-store-and-query.md)）
- **持久化 Model + 具名 Repository**（`Persistence/AuditLogs/AuditLog.cs` + `IAuditLogRepository.cs`）：字段对齐 [ADR-008 §决策](../../03-architecture/adr/ADR-008-audit-log-store-and-query.md) 表结构关键字段，`ListAuditLogs` 直接复用 [HD-007 `AuditLogQuery`](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#34-auditauditlogquerycs) 作为过滤参数，不重复定义过滤类型

| Q7  | `DefaultAuditLogger` / `AuditLogWriteBackgroundService` / `AuditLogRetentionCleanupBackgroundService` 均通过 `IServiceScopeFactory.CreateScope()` 逐次解析 `IPersistenceProvider`，**不**在构造函数直接注入 `IPersistenceProvider`                                                                                                         | `IPersistenceProvider` 以 `AddScoped` 注册（[HD-009 §7](../Inkwell.Persistence.EFCore/HD-009-Inkwell.Persistence.EFCore-base.md) `services.AddScoped<IPersistenceProvider, EfCorePersistenceProvider>()`），而 `DefaultAuditLogger`（`AddSingleton<IAuditLogger, DefaultAuditLogger>()`）与两个 `BackgroundService` 派生类（`AddHostedService`）均以 Singleton 生命周期注册——直接构造函数注入会构成 Singleton 消费 Scoped 依赖的 DI 反模式，与本仓库已实证过的 [HD-010 B16/C96](../design-review-report.md#b16inmemoryrowversioninterceptor-以错误的-di-服务类型注册导致rowversion-拦截器永不执行c96) 同根因风险类别（[design-review-report.md §28 B-1](../design-review-report.md)）；用 `IServiceScopeFactory` 逐次创建作用域是标准规避方式                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

| Q9  | Channel 类型选 `Channel<AuditLog>`（已完成 `AuditLogRequest → AuditLog` 映射后的领域对象），**不**是 `Channel<AuditLogRequest>`                                                                                                                                                                                                            | 让 `AuditLogWriteBackgroundService` 完全不需要感知 `AuditLogRequest`/`AuditContext` 的映射细节，只处理"持久化一个已经组装好的 `AuditLog` + 重试 + fallback"这一件事，职责更单一；映射逻辑集中在 `DefaultAuditLogger.LogAsync` 一处（详见 §3.5）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Q10 | Channel 用 `Channel.CreateUnbounded`（无界），**不**设容量上限                                                                                                                                                                                                                                                                             | [HD-007 §7.1](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#71-性能预算)"`LogAsync` facade overhead P50 < 10ms"要求 `LogAsync` 几乎不阻塞；有界 Channel 在满载时 `WriteAsync` 会 `await` 直至有空位，可能拖慢业务调用方；无界 Channel 保证 `LogAsync` 恒定快速返回，代价是消费者持续故障时内存增长，由磁盘 fallback（3 次重试后触发）及时止损                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

### 1.4 与 HD-007 的边界声明（本 HD 是唯一实现方，非查询专属实现方）

已 reviewed 的 [HD-007 §6 Builder DSL 钩子](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#6-builder-dsl-钩子给-inkwellcoreauditlogs-hd-的契约) 原文即以本 HD 为目标读者，逐字写明 `AddDefaultAuditLogger()` 扩展方法**必须**：

1. 调用 `builder.Services.AddSingleton<IAuditLogger, DefaultAuditLogger>()`
2. 注册 `IValidateOptions<AuditLoggerOptions>`
3. 无需与 `InkwellProvidersOptions` 做交叉校验
4. 返回 `builder`

本 HD 严格按此契约实现（详见 [§3.9](#39-inkwellcoreauditlogsauditloggerbuilderextensionscs)），并额外补齐 HD-007 §11 明确留白的三项：写入重试队列、磁盘 fallback、保留期清理后台任务——这三项是 HD-007 起草时就**已经预见并显式指派**给本 HD 的工作项，不是本 HD 擅自扩大范围。

## 4. BCL 异常与日志（补充 HD-001 §4 / HD-002 §4 / HD-007 §4）

### 4.1 错误码

本 HD 不分配 `INK-AUDITLOGS-NNN` 错误码，延续全仓 [ADR-023](../../03-architecture/adr/ADR-023-port-signature-bare-task-with-exceptions.md) 最终态规约——错误语义全部走 BCL 异常类型 + OTel `exception.*` 五字段。

### 4.2 BCL 异常分类

- **`DefaultAuditLogger.LogAsync`**：完全遵循 [HD-007 §4.2](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#42-bcl-异常分类业务失败-vs-程序错误) 已锁定分类，本 HD 不新增分支
- **`DefaultAuditLogger.QueryAsync`**：同上，`IAuditLogRepository.ListAuditLogs` 抛出的 `IOException`/`TimeoutException` 原样透传
- **`AuditLogWriteBackgroundService`/`AuditLogRetentionCleanupBackgroundService`**：全部异常在服务内部捕获记录，不向 .NET Generic Host 传播（避免单次持久化/清理故障导致整个宿主进程的托管服务集合终止）

### 4.3 OTel span / 字段

沿用 [HD-007 §4.3](../Inkwell.Abstractions/HD-007-Inkwell.Abstractions-audit-logger-port.md#43-otel-span--字段) 已锁定的 `audit.log`/`audit.log.retry`/`audit.log.fallback_write`/`audit.query` 四个 span 及全部字段；本 HD 新增 1 个 span：`audit.retention_cleanup`（[§3.8](#38-inkwellcoreauditlogsauditlogretentioncleanupbackgroundservicecs)）。
