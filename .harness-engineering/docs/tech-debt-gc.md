# 熵与技术债务 GC

本文件是 Harness Engineering 规范的技术债务治理章节。在进入操作细节之前，先把两个常被忽略的问题讲清楚：**交付后真的不需要改代码吗？** **"熵"和"GC"这两个词到底什么意思？** 然后再给黄金原则、定期 GC 任务、使用原则三块操作。

> 该章实践经验来自 OpenAI Codex 团队在 *Harness engineering* 一文中提出的方案。

## 1. 为什么交付后还要做 GC

常见误区是"项目交付后基本不需要改代码，除了需求变更或修 bug"。这个判断在**短生命周期项目**（一次性活动页、毕设、demo）里成立，但任何要**长期运营**的系统都不成立：

- Lehman 软件演化定律：持续使用的系统必须持续修改，否则功能逐渐降级
- 行业经验：维护占整个软件生命周期成本的 **60–80%**，开发只占 20–40%
- 一线观察：上线后第二年的 commit 频率往往**高于**开发期

也就是说，"交付"是真正运营的开始，不是终点。维护期的改动可以归为三类，**只有其中一类才是"需求变更或修 bug"**：

### A. 外部环境驱动（世界变了）

依赖升级（.NET / 第三方包大版本切换）、安全漏洞披露（log4j、Spring4Shell、xz backdoor、npm 链式 CVE）、第三方 API 改版（微信支付 / Stripe webhook）、平台政策（iOS 强制 IPv6、Android 后台限制、GDPR / 个保法合规）、协议吊销（TLS 1.0/1.1、SHA-1 证书）……这些既不是需求变更也不是 bug，是外部 deadline 砸到头上。

### B. 内部环境驱动（系统自己要养）

数据增长（SQLite 撑一年后必须迁 PG）、流量增长（单实例要拆分布式）、新功能 piggyback（老模块要复用必先重构）、监控冒头的隐患（慢查询 / 内存泄漏 / 接口 P99 超标）。

### C. 文档与决策腐败 ★ GC 的主战场

| 现象 | 普遍程度 |
| --- | --- |
| API 改了 v1.2，README 还是 v1.0 的样子 | 90% 项目都中招 |
| DB 加字段，`docs/04-detailed-design/` 没同步回写 | 高频 |
| 新人 onboarding 时看的 docs，多数已过时 | 普遍 |
| 评审当时拍板的决策，半年后再翻找不到 | 普遍 |

**关键洞察**：A、B 两类有 lint / CI / 监控 + 人盯能压住；**C 类几乎不可能在 PR review 阶段发现**——评审只看新增 diff，不会主动扫"已存在但开始腐烂"的文档与规范。这是 GC 必须存在的根本理由。

## 2. 术语：熵 与 GC

两个都是借喻。

**熵（entropy）** 来自热力学：没人维护的系统会自发地越来越乱。仓库的"熵增"长这样——代码逐渐偏离最初设计原则、文档与实际行为不一致、"先 hack 一下下个迭代再修"的临时方案变永久、TODO 越积越长没人回头。仓库不会自我修复，就像房间不会自己变干净。

**GC（Garbage Collection · 垃圾回收）** 来自 Java / Go / C# / Python 等语言运行时：周期性扫描内存、清理不再被引用的对象。本文借用这个比喻：

| 运行时 GC | Harness 里的"技术债务 GC" |
| --- | --- |
| 扫内存对象 | 扫仓库代码 + 文档 |
| 标记不再被引用 | 标记偏离黄金原则的代码 / 与代码不一致的过时文档 |
| 自动清理 | 让 doc-gardener Agent 开 PR 修复 |
| 周期性运行 | 每周 / 每个 sprint 跑一次 |

GC 的执行者是 [`hx-DocGardener`](../agents/doc-gardener/AGENT.md)。Agent 名字用 **gardener（园丁）** 而不是 **builder（建设者）**——对应的是修剪 / 除草 / 浇水这种**维护性**动作，不是**新建**。

### 为什么 H1–H6 阶段 Agent 不够用

H1–H6 是开发期一次性走完的流程；交付后没人会再触发它们。但腐败是**持续**发生的，所以必须有一个**横切、长期常驻**的 agent。这就是 11 个 agent 里只有 `hx-DocGardener` 用 `hx-` 前缀的原因——它不属于任何阶段，是阶段之外的常驻反馈机制。后续这条 `hx-*` 线还可能扩展：`hx-dependency-watcher`（依赖与 CVE 监控）、`hx-perf-regression-detector`（性能告警）、`hx-security-scanner`（安全扫描）等，**它们全都是交付后才真正发挥价值的 agent**。

## 3. 黄金原则（Golden Principles）

团队应提炼出一组可机械化检查的"黄金原则"，描述项目期望代码库保持的形状：

- 优先使用共享工具包，避免手写重复逻辑
- 在边界始终使用类型验证（parse, don't validate），不凭猜测推送数据形状
- 结构化日志、命名约定、文件大小上限等"品味不变式（taste invariants）"需以 Lint 硬拦截
- 跨层依赖只能沿架构图预设方向，逾越者报错

这些原则需写进 `docs/` 下的权威文档（如 `quality-grade.md`）并同步编码为可执行检查。

> 与 [`../README.md` 第 6.5 节](../README.md#65-软约束的失败模式与处置阶梯) 软约束阶梯口径一致：能机械判定的原则一律下沉为 Scripts / Lint / CI；停在自然语言层的原则迟早会被解释性绕过。

## 4. 定期 GC 任务

按第 1 节的三类触发源，在仓库里配置定期运行的后台 AI 任务做对应清理：

- **应对 A 类**（外部环境）：扫描第三方依赖的 CVE 公告与版本变更，开启升级 PR
- **应对 B 类**（内部环境）：扫描代码库与黄金原则的偏离 / 慢查询日志 / TODO 标记，开启重构 PR；持续更新 `docs/06-implementation/exec-plans/tech-debt-tracker.md` 中的未完成项
- **应对 C 类**（文档腐败 · 主战场）：扫描 `docs/` 下与代码实际行为不一致的过期文档（doc-gardening），开启修复 PR
- 合并选项：质量评级 / quality grade 表可在项目初期仅补充到 `docs/04-detailed-design/` 或 `docs/07-release/` 中

文档治理这条线由 [`../agents/doc-gardener/AGENT.md`](../agents/doc-gardener/AGENT.md) 负责落地。

## 5. 使用原则

- **持续偿还 > 集中重构**：技术债务像高利息贷款，每日少量偿还远优于积压后被迫集中返工。
- **人的品味一次捕获，机器永久执行**：评审心得、重构经验、线上故障复盘，要么转化为 `AGENTS.md` / Skill 里的指导，要么转化为 Lint / Hooks / CI 检查。
- **允许小幅 PR 自动合并**：GC 产出的 PR 如果可以在一分钟内评审完毕，应设置成可自动合并。
