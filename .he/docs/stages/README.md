# 各阶段细则（H1–H6）

本目录是 Harness Engineering 规范的**阶段细则部分**。每个阶段单独一份文件，单文件可独立阅读、独立给 Agent 加载，避免一次性吃下整份 500 行的合订本。

> 阶段编号 `H1`–`H6` 中的 `H` 取自 *Harness*，`Hn` 即 **Harness Stage n** 的缩写。完整定义见 [`../../README.md` 的"标准流程"小节](../../README.md#3-标准流程)。

| H1   | 需求、UI 与交互原型 | [`h1-requirements-and-prototype.md`](./h1-requirements-and-prototype.md) | `docs/01-requirements/` + `docs/02-prototype/` + `prototypes/` |

每份阶段文件都采用相同的本地章节结构，便于跨阶段对照：

| 本地章节 | 内容              | 备注                                          |
| -------- | ----------------- | --------------------------------------------- |
| §1       | 阶段定位与目标    | 为什么有这个阶段、它解决什么问题              |
| §2       | 输入              | 上游需要交付哪些文档/资料才能进入本阶段      |
| §3       | 输出物            | 本阶段必须产出的文件清单与建议路径            |
| §4       | 必须覆盖/必含字段 | 本阶段产物的最小完备性要求（机械化清单）      |
| §5       | 扩展要求          | 阶段特有的额外约束（如文件级设计、UI 必含）    |
| §6       | 评审门禁          | 进入下一阶段前必须 PASS 的检查项              |
| §7       | （仅 H5）提交要求 | H5 因多一节"提交规范"，单独保留 §7            |

> 引用规范：从其他文档引用某阶段的子章节，统一使用 `stages/h1-requirements-and-prototype.md §5`、`stages/h3-detailed-design.md §4` 这种形式。**不再使用** `stages.md 第 4.5 节` 的旧式跨文件全局编号。

## 2. 跨阶段流程：失败回退路径

H1–H6 是单向流水线，但下游一定会发现上游遗漏。本节定义"在哪个阶段发现什么问题，应该回退到哪个阶段"，避免出现两类反模式：① 出问题就推倒重来；② 在当前阶段就地"擦屁股"绕过上游。

| 发现失败的位置                  | 失败现象（举例）                                                          | 回退到 | 回退后必须做什么                                                              |
| ------------------------------- | ------------------------------------------------------------------------- | ------ | ----------------------------------------------------------------------------- |
| H2 架构选型                     | 选不出能落地的方案 / 关键非功能需求（性能、合规）在需求里没明确           | H1     | 补需求中"非功能需求"与"权限/数据边界"两节，把缺失项作为新需求重新评审          |
| H3 详细设计                     | 设计文件无法对齐架构 / 接口契约和需求里的字段不一致                       | H2     | 修正 `architecture.md` / `api-design.md` 边界，必要时补一条 ADR                |
| H4 测试用例                     | 某条需求找不到可测试的验收标准 / 设计层缺接口契约导致写不出 TC            | H1 或 H3 | 缺验收标准 → H1；缺接口契约 → H3。回退后该需求/设计的 status 必须重新走评审 |
| H5 编码（实现阶段）             | 设计中遗漏了某文件的依赖 / 数据结构 / 错误处理                            | H3     | 修详细设计；不在编码任务卡里就地编造，避免架空设计文档                        |
| H5 编码（测试阶段）             | 写不出测试 / 测试始终失败且非代码 bug（而是用例本身设计错）                 | H4     | 修测试用例；编码 Agent 不得擅自"修改测试以让它通过"                            |
| H5 提交评审                     | `Design / Tests / Task` 三件套追溯不上                                    | H5 起点 | 重新生成任务卡（确保上游引用齐备），不直接放行提交                              |
| H6 运行验证                     | 实际行为与设计不符（设计错）                                              | H3     | 改 `detailed-design.md` 并在 `release-notes.md` / `traceability-matrix.md` 写明 |
| H6 运行验证                     | 实际行为与需求不符（需求错或漏）                                          | H1     | 改 `requirements.md`，可能触发后续阶段连锁更新                                  |
| 任意阶段                        | Reviewer 反复打回超过 [`io-contracts.md`](../../agents/_shared/io-contracts.md) 的反问预算上限 | 当前阶段起点 | 返回 `blocked`，把"已收敛点 / 仍未达成共识点"写入 `task-board.md` 等待人工裁决 |

约束：

- **禁止在下游就地修复上游缺陷**。例如 H5 编码时发现设计漏字段，必须回退到 H3 改设计稿后再回到 H5，不得在代码里"顺手补上"而设计文档不动。
- **回退必须留下凭证**。改动哪一份上游产物，写入对应阶段的 review 纪要或 commit message，让追溯链完整可见。
- **回退不重置已完成的下游产物**。如 H3 修了一个字段，影响范围内的 H4 测试用例和 H5 代码同步修，影响范围外的不动——保持小步迭代。

## 3. 阅读路径建议

- **首次读规范**：先看 [`../../README.md`](../../README.md)，再按 H1→H6 顺序逐文件浏览。
- **执行某个阶段**：直接打开对应阶段文件即可（如做 H3 评审就只读 `h3-detailed-design.md` + 本文件 §2 失败回退）。
- **写 Agent / Skill 引用**：从 `agents/` 引用阶段细则时，使用稳定路径 `docs/stages/h<N>-<slug>.md §<本地章节号>`。
